  - `SKILL.md`（工厂）→ 引导用户创建前任

## 阶段 2：Skill 的入口文件 — SKILL.md 结构解析

### 学习目标
掌握 SKILL.md 的完整结构，理解每个部分的作用，能独立写出一个合法的 SKILL.md。

### 需要掌握的内容

**Frontmatter 字段**
```yaml
---
name: skill-name          # Skill 的唯一标识符
description: ...          # 一句话描述，影响 Claude 是否触发
argument-hint: [args]     # 命令行参数提示
version: 1.0.0
user-invocable: true      # 是否允许用户通过 /name 调用
allowed-tools: Read, Write, Edit, Bash  # 声明可用工具
---
```

**主体结构的四个核心部分**
1. **触发条件** — 定义什么情况下激活这个 Skill
2. **工具使用规则** — 声明每类任务用哪个工具
3. **安全边界** — 明确 Claude 不能做什么
4. **主流程** — 分步骤描述 Claude 的执行逻辑

**触发条件的两种写法**
- 精确命令：`/create-ex`
- 自然语言模式：`"帮我创建一个前任 skill"`

### 阅读源码
| 文件 | 阅读重点 |
|------|----------|
| `SKILL.md`（完整文件） | 整体结构、各部分的写法 |
| `SKILL.md`（工具使用规则表格） | 工具分配的决策逻辑 |
| `SKILL.md`（安全边界部分） | 如何用自然语言约束 AI 行为 |

### 思考练习
> 对比"触发条件"和"工具使用规则"两个部分：前者是告诉 Claude **何时**行动，后者是告诉 Claude **如何**行动。你能在自己的 Skill 设计中区分这两类指令吗？

## 阶段 3：用户交互设计 — 信息收集流程

### 学习目标
学会设计结构化的用户信息收集流程，理解"必填/可选"、"汇总确认"等交互模式。

### 需要掌握的内容

**信息收集的设计原则**
- 最小必要信息：只问真正需要的（3 个问题，不是 30 个）
- 渐进式收集：先问必填，再问可选
- 示例驱动：每个问题都给出具体示例，降低用户认知负担
- 汇总确认：收集完后展示摘要，让用户有机会纠正

**字段解析设计**
- 如何从一句自然语言中提取结构化字段
- 例：`"在一起两年 分手半年 互联网产品经理"` → `{together_duration, apart_since, occupation}`

**Slug 生成规则**
- 中文 → 拼音，英文 → 小写，空格 → 下划线
- 作为文件系统路径的安全标识符

### 思考练习

## 阶段 4：Prompt 工程（分析类）— 从原材料提取结构化信息

### 学习目标
掌握"分析型 Prompt"的写法：如何引导 Claude 从非结构化原材料中提取有价值的结构化信息。

### 需要掌握的内容

**分析型 Prompt 的核心要素**
1. **任务声明**：明确告诉 Claude 要做什么
2. **提取维度**：列出所有需要提取的字段/维度
3. **输出格式**：规定输出的 Markdown 结构
4. **注意事项**：处理边界情况（数据冲突、信息缺失等）

**双轨分析模式**
- Track A（Memory）：提取客观事实 — 时间线、地点、事件
- Track B（Persona）：提取行为模式 — 说话风格、情感表达、依恋类型
- 两条线独立分析，避免互相干扰

**从聊天记录中提取的维度**
- 语言层：口头禅、标点习惯、emoji 偏好
- 行为层：回复速度、主动频率、话题分布
- 情感层：依恋类型、爱的语言、冲突模式

### 阅读源码
| 文件 | 阅读重点 |
|------|----------|
| `prompts/memory_analyzer.md` | 8 个提取维度的设计、输出格式模板 |
| `prompts/persona_analyzer.md` | 说话风格、情感模式、依恋类型的提取方法 |
| `SKILL.md`（Step 3 部分） | 如何在主流程中调用这两个分析器 |

### 思考练习
> `memory_analyzer.md` 中有一条注意事项："聊天记录中的事实优先于用户口述"。为什么要这样设计？这体现了什么样的数据可信度判断？

## 阶段 5：Prompt 工程（生成类）— 构建分层 Persona 模型

### 学习目标
理解"生成型 Prompt"的写法，重点掌握分层结构设计和标签翻译机制。

### 需要掌握的内容

**5 层 Persona 结构**
```
Layer 0 — 硬规则（不可覆盖）
Layer 1 — 身份锚点（基本信息）
Layer 2 — 说话风格（语言表达）
Layer 3 — 情感模式（内心世界）
Layer 4 — 关系行为（互动方式）
```

**层级优先级的意义**
- Layer 0 是"宪法"，任何下层规则都不能违反它
- 越往下越具体，越容易被用户纠正
- 这种设计防止 AI 角色"崩人设"

**标签翻译机制**
将用户输入的模糊标签翻译为具体行为规则：
- `"话痨"` → 消息密度高、话题跳跃快、不等对方回复就继续发
- `"嘴硬心软"` → 表面说"随便"但实际在意、吵架不先道歉但会用行动弥补
- `"冷暴力"` → 生气时忽略消息数小时甚至数天、需要对方主动破冰

**Memory Builder 的模板设计**
- 如何把分析结果填入固定模板
- 占位符 `{xxx}` 的使用

### 阅读源码
| 文件 | 阅读重点 |
|------|----------|
| `prompts/persona_builder.md` | 5 层结构的完整模板、Layer 0 硬规则的写法 |
| `prompts/memory_builder.md` | 关系记忆的模板结构 |
| `prompts/persona_analyzer.md`（标签翻译表部分） | 20+ 个标签的翻译规则 |

### 思考练习
> 为什么 Layer 0 要包含"不说ta在现实中绝不可能说的话"这条规则？如果没有这条规则，AI 角色会出现什么问题？

## 阶段 6：结构化数据输出 — meta.json 与文件组织

### 学习目标
掌握如何设计 Skill 的元数据结构，理解文件系统组织的最佳实践。

### 需要掌握的内容

**meta.json 的设计**
```json
{
  "name": "...",
  "slug": "...",
  "created_at": "ISO时间",
  "updated_at": "ISO时间",
  "version": "v1",
  "profile": { ... },
  "tags": { ... },
  "memory_sources": [...],
  "corrections_count": 0
}
```

**字段设计原则**
- 时间用 ISO 8601 格式（机器可读）
- slug 作为唯一标识符（文件系统安全）
- `corrections_count` 追踪进化次数（可观测性）
- `memory_sources` 记录数据来源（可溯源性）

**目录结构设计**
```
exes/{slug}/
├── SKILL.md          # 可运行的完整 Skill
├── memory.md         # Part A：关系记忆
├── persona.md        # Part B：人物性格
├── meta.json         # 元数据
├── versions/         # 历史版本备份
└── memories/         # 原始素材
    ├── chats/
    ├── photos/
    └── social/
```

**关注点分离原则**
- memory.md 和 persona.md 分开存储，各自独立进化
- SKILL.md 是两者的合并视图，由工具自动生成

### 阅读源码
| 文件 | 阅读重点 |
|------|----------|
| `SKILL.md`（Step 5 写入文件部分） | meta.json 的完整结构、目录创建命令 |
| `tools/skill_writer.py` | 如何用代码管理 Skill 文件的读写和合并 |

### 思考练习
> 为什么要把 memory.md 和 persona.md 分开，而不是直接写一个大文件？这种"关注点分离"在软件工程中有什么对应的设计模式？

## 阶段 7：工具集成 — Python 脚本与 Bash 调用

### 学习目标
理解 Claude Code Skill 如何通过 Bash 工具调用外部脚本，掌握工具集成的设计模式。

### 需要掌握的内容

**Claude Code 的工具体系**
| 工具 | 用途 |
|------|------|
| `Read` | 读取文件（含图片、PDF） |
| `Write` | 创建新文件 |
| `Edit` | 修改已有文件 |
| `Bash` | 执行 shell 命令（调用 Python 脚本） |

**通过 Bash 调用 Python 脚本的模式**
```bash
python3 ${CLAUDE_SKILL_DIR}/tools/wechat_parser.py \
  --file {path} \
  --target "{name}" \
  --output /tmp/wechat_out.txt \
  --format auto
```

**脚本设计原则**
- 使用 `argparse` 定义清晰的命令行接口
- 输出到临时文件（`/tmp/`），Claude 再用 Read 读取
- 支持 `--format auto` 自动检测输入格式
- 错误时输出有意义的错误信息

**wechat_parser.py 的核心逻辑**
- 多格式支持（txt/html/json/csv）
- 提取：口头禅、emoji 偏好、回复速度、话题分布

### 阅读源码
| 文件 | 阅读重点 |
|------|----------|
| `tools/wechat_parser.py` | argparse 接口设计、多格式解析逻辑 |
| `tools/skill_writer.py` | `--action list/init/combine` 的接口设计 |
| `tools/photo_analyzer.py` | EXIF 数据提取（Pillow 库的使用） |
| `SKILL.md`（工具使用规则表格） | 工具分配决策的完整映射 |

### 思考练习
> 为什么解析微信记录要用 Python 脚本，而不是直接让 Claude 读取原始文件？什么情况下应该用脚本处理，什么情况下直接用 Claude 的 Read 工具？

## 阶段 8：状态管理与版本控制

### 学习目标
理解如何在 Skill 中实现状态持久化和版本管理，掌握"备份优先"的设计思想。

### 需要掌握的内容

**版本管理的必要性**
- Skill 会随用户反馈持续进化
- 每次更新前必须备份，支持回滚
- 版本号格式：`v1`, `v2`, ... 或时间戳

**version_manager.py 的核心操作**
```bash
# 备份当前版本
python3 tools/version_manager.py --action backup --slug {slug} --base-dir ./exes

# 回滚到历史版本
python3 tools/version_manager.py --action rollback --slug {slug} --version v2 --base-dir ./exes

# 列出所有版本
python3 tools/version_manager.py --action list --slug {slug} --base-dir ./exes
```

**备份策略**
- 备份存储在 `exes/{slug}/versions/` 目录
- 文件命名：`{timestamp}_{version}/`
- 备份内容：memory.md + persona.md + meta.json

**管理命令的设计**
- `/list-exes`：列出所有已创建的前任
- `/ex-rollback {slug} {version}`：回滚
- `/delete-ex {slug}` / `/let-go {slug}`：删除（含确认步骤）

### 阅读源码
| 文件 | 阅读重点 |
|------|----------|
| `tools/version_manager.py` | backup/rollback/list 三个操作的实现 |
| `SKILL.md`（管理命令部分） | 命令设计和对应的 Bash 调用 |
| `SKILL.md`（进化模式：追加记忆，第 4 步） | 更新前自动备份的流程 |

### 思考练习
> `/delete-ex` 和 `/let-go` 是同一个操作，但输出不同。这种"别名命令"的设计体现了什么？在你的 Skill 设计中，什么时候值得为同一功能提供多个触发词？

## 阶段 9：进化机制 — 增量更新与对话纠错

### 学习目标
掌握 Skill 的"持续进化"设计模式：如何处理新增信息和用户纠错，而不破坏已有内容。

### 需要掌握的内容

**两种进化模式**

**模式 1：追加记忆（Append）**
- 触发：用户提供新的聊天记录/照片/回忆
- 原则：只追加，不覆盖；冲突标记 `[⚠️ conflict]`
- 流程：读取现有文件 → 分析增量 → 备份 → Edit 追加 → 重新生成 SKILL.md

**模式 2：对话纠错（Correction）**
- 触发：用户说"不对"/"ta不会这样说"
- 原则：不删除原内容，而是追加 Correction 记录
- 纠错记录格式：
  ```markdown
  ## Correction 记录
  ### Correction #1
  - 原内容：...
  - 纠正为：...
  - 用户说明：...
  - 时间：...
  ```
- 原文标注：`[已纠正，见 Correction #1]`

**merger.md 的增量分析逻辑**
- 对比新旧内容，识别新增信息
- 判断是否与现有内容冲突
- 决定追加位置

### 阅读源码
| 文件 | 阅读重点 |
|------|----------|
| `prompts/merger.md` | 增量分析的完整逻辑 |
| `prompts/correction_handler.md` | 纠错识别和记录格式 |
| `SKILL.md`（进化模式两个部分） | 两种进化模式的完整流程 |

### 思考练习
> 为什么纠错时要"追加 Correction 记录"而不是直接修改原文？这种"不可变历史"的设计思想在哪些软件系统中也有体现？

## 阶段 10：安全边界、工程化收尾与完整复盘

### 学习目标
理解 Skill 的安全设计原则，掌握工程化的完整交付标准，并对整个项目做系统性复盘。

### 需要掌握的内容

**安全边界的设计**
- 用途限制：明确声明 Skill 的使用场景
- 行为禁止：列出 Claude 绝对不能做的事
- 心理健康保护：识别不健康使用模式并温和干预
- Layer 0 硬规则：AI 角色的"宪法级"约束

**工程化交付标准**
一个完整的 Skill 项目应包含：
- [ ] `SKILL.md`：入口文件，结构完整
- [ ] `prompts/`：各步骤的详细 Prompt 模板
- [ ] `tools/`：数据处理脚本（如需要）
- [ ] `docs/`：PRD、使用指南
- [ ] `README.md`：安装和使用说明
- [ ] `.gitignore`：排除用户数据（`exes/`）

**双语支持的实现**
- 在 SKILL.md 开头声明语言检测规则
- 中英文版本并列，结构相同

**整体架构复盘**
```
用户输入
    ↓
SKILL.md（编排层）
    ↓
prompts/*.md（Prompt 模板层）
    ↓
tools/*.py（数据处理层）
    ↓
exes/{slug}/（输出层）
    ├── SKILL.md（可运行产物）
    ├── memory.md
    ├── persona.md
    └── meta.json
```

### 阅读源码
| 文件 | 阅读重点 |
|------|----------|
| `SKILL.md`（安全边界部分） | 5 条安全规则的写法 |
| `docs/COMPLIANCE.md` | 合规性设计 |
| `SKILL.md`（生成的 SKILL.md 结构，Step 5） | 最终产物的"运行规则"部分 |
| 整个项目目录 | 工程化交付的完整结构 |
